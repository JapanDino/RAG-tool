## ЧТО СДЕЛАНО ХОРОШО ✅

### Архитектура и инфраструктура
- **Провайдер-абстракция для эмбеддингов** — чистый ABC-интерфейс с 4 реализациями (`hash`, `local`, `openai`, `random`). Замена модели не требует правки бизнес-логики.
- **Двойной режим async/sync** — система работает без Celery (`ENABLE_CELERY=0`) и с ним. Удобно для разработки и деплоя.
- **Fallback-цепочка экстракторов** — `NatashaNerExtractor` → `HeuristicExtractor` с graceful degradation.
- **Laplace-сглаживание в Bloom-классификаторе** — формула `(c+1)/(total+6)` корректно исключает нулевые вероятности.
- **Drift-коррекция суммы вектора** — вектор вероятностей всегда суммируется ровно в 1.0.

### Качество NLP-компонентов
- **Natasha NER с MorphVocab** — нормализация русских словоформ перед дедупликацией. Правильная работа с морфологией.
- **Символьные офсеты** (`sentence_idx`, `char_start`, `char_end`) — полная прослеживаемость узлов к исходному тексту.
- **Гибридный классификатор** — keyword-baseline + опциональный LLM (`gpt-4o-mini`) с graceful fallback при ошибке API.
- **Pydantic-валидация LLM-вывода** — `LLMClassifyOut` проверяет shape `prob_vector` (`min_length=6, max_length=6`).
- **Метрики качества** — `evaluate.py` реализует micro-F1, macro-F1, Hamming Loss вручную и корректно.

### База данных
- **Dual-storage** — `embeddings` (chunk-уровень) + `knowledge_nodes.vec` (node-уровень). Правильное разделение.
- **`embedding_model` колонка** — все векторные запросы фильтруются по модели. Кросс-провайдерные сравнения заблокированы.
- **CASCADE DELETE** на `dataset_id` — удаление датасета чистит все зависимые данные.
- **`model_info` JSONB** — полная провенанс-информация: тип экстрактора, классификатора, позиция в тексте, rationale.
- **Upsert с `version`** — идемпотентная запись аннотаций через `ON CONFLICT ... DO UPDATE`.

### Фронтенд — UX и полировка
- **Toast-система** с 3 типами, auto-dismiss 3.8s, click-to-dismiss. Профессиональный уровень.
- **Onboarding overlay** при первом визите, 4 шага — хорошо написан, уместно краткий.
- **FlowStep-индикатор прогресса** на вкладке Анализа — пользователь видит 4 шага воркфлоу и их состояние.
- **Keyboard shortcuts** — Alt+1-5 (вкладки), 1-6 (уровни), Enter (сохранить лейбл), Ctrl+Z (отмена). Labeling-воркфлоу полностью клавиатурный.
- **Undo-стек** 10 уровней для разметки — защита от случайных кликов.
- **Skeleton-лоадеры** в гриде узлов — не просто спиннер, card-shaped плейсхолдеры.
- **История текстов** (5 последних) в localStorage — продуктивная деталь.
- **Confidence band** (high/medium/low) и `reviewHint` для узлов с низкой уверенностью — академически обосновано.
- **Логарифмический размер узлов** — правильный выбор для skewed-распределений частоты.
- **Форма узла как второй визуальный канал** — помогает пользователям без цветовосприятия.
- **Множественный экспорт** (JSON, CSV, JSONL) — уместно для исследовательского инструмента.
- **Светлая и тёмная тема** — CSS custom properties, persisted preference.

## ЧТО НЕ В ПОРЯДКЕ ❌

### КРИТИЧЕСКИЕ ПРОБЛЕМЫ 🔴

#### [C-1] Провайдер эмбеддингов по умолчанию — `hash` (семантически бессмысленный)
**Файл:** `backend/app/services/embedding_provider.py:175`  
`os.getenv("EMBEDDING_PROVIDER", "hash")` — при отсутствии переменной используется хеш-провайдер. HashingProvider — это TF-IDF bag-of-words, он не кодирует семантику. Косинусная близость двух hash-векторов равна пересечению токенов, а не смысловой близости. При этом `.env.example` правильно ставит `local`, но если модель не загрузится, система молча откатится на `hash` без видимого предупреждения пользователю.

#### [C-2] Чанкование — фиксированный срез по 800 символов без перекрытия
**Файл:** `backend/app/tasks/tasks.py:313`  
```python
parts = [text_str[i:i+800] for i in range(0, len(text_str), 800)]
```
Это худший возможный способ чанкования для семантической системы:
- Предложения и концепты разрезаются посредине
- Нулевое перекрытие — концепт на границе невидим обоим чанкам
- 800 символов ≈ 100–130 токенов для русского, намного меньше контекстного окна модели
- В проекте есть `chunking.py` с sentence-aware сплиттером — он полностью игнорируется

#### [C-3] Bloom-аннотация: `score = длина текста`
**Файл:** `backend/app/utils/bloom.py:12-24`  
```python
score = min(1.0, 0.5 + len(chunk.strip())/2000.0)
```
Скор аннотации зависит только от длины. 500-символьный факт («Знать») и 500-символьная задача синтеза («Создавать») получают одинаковый скор. При `ENABLE_LLM=0` (default) поле `score` в `BloomAnnotation` полностью бессмысленно. Фильтрация по `min_score` в endpoints статистики — вводит в заблуждение.

#### [C-4] Колонка `vec` отсутствует в SQLAlchemy ORM
**Файл:** `backend/app/models/models.py`  
`vec vector(1536)` используется во всех raw SQL-запросах, но не объявлена в моделях. Комментарий «создадим в SQL-миграции» означает: существование индекса зависит от ручного запуска скрипта. `alembic autogenerate` колонку не видит и может её удалить. Нет гарантии, что HNSW/IVFFlat индекс был создан.

#### [C-5] `reindex_nodes` блокирует HTTP-воркер
**Файл:** `backend/app/routers/datasets.py:107-138`  
Для 500 узлов с local sentence-transformer — это 30–120 секунд на CPU в синхронном HTTP-хендлере. FastAPI блокирует Uvicorn-воркер на всё время.

#### [C-6] Дублирование типов между `index.tsx` и `GraphView.tsx`
`BloomLevel`, `BLOOM_LEVELS`, `LEVEL_LABELS`, `LEVEL_COLORS`, `AnalyzeNode` определены независимо в двух файлах. Изменение цвета/лейбла в одном файле не применяется к другому — молчаливая рассинхронизация.

#### [C-7] Нет responsive-дизайна
`home.module.css` — жёсткий трёхколоночный лейаут без media queries. На экранах < 1100px горизонтальный overflow или нечитаемый коллапс.

### СЕРЬЁЗНЫЕ ПРОБЛЕМЫ 🟠

#### [M-1] Regex-баг при разбиении на предложения
**Файл:** `backend/app/services/node_extractor.py:13`  
```python
SENT_SPLIT_RE = re.compile(r"[.!?\\n]+")
```
`\\n` в raw-строке — это литеральный обратный слеш + `n`, а не перенос строки. Параграфы, разделённые `\n`, не разбиваются на предложения → длинные «предложения» → плохой трекинг офсетов.  
Правильно: `r"[.!?\n]+"`.

#### [M-2] Padding до 1536 измерений — несовместимые пространства
**Файл:** `backend/app/services/embedding_provider.py:29-39`  
`multilingual-e5-small` → 384 dims, pad до 1536. При переключении на OpenAI (нативные 1536 dims) векторы физически несовместимы, хотя `embedding_model` это блокирует. Ситуация не задокументирована и создаёт путаницу.

#### [M-3] Классификация работает на контексте ≤ 240 символов или просто на заголовке
**Файл:** `backend/app/routers/analyze.py:127`  
```python
text_for_cls = node.get("context_snippet") or node["title"]
```
Bloom-классификация одного слова или одного предложения — почти случайный результат. Нужен контекст 500–1000 символов вокруг позиции узла.

#### [M-4] Два разных Bloom-классификатора с разной семантикой
- `bloom_classifier.py` (`POST /analyze`): 5 глаголов на уровень, нет Laplace-сглаживания
- `bloom.py` (основной pipeline): полный список из JSON, Laplace-сглаживание

Одинаковый текст через разные endpoints → разные `prob_vector`. Несоответствие ТЗ.

#### [M-5] Несогласованные метрики расстояния: `<=>` (cosine) vs `<->` (L2)
`tasks.py`, `nodes.py` — L2; `search.py`, `graph.py` — cosine. Для единичных векторов оба монотонно эквивалентны, но `1 - L2_distance` — это не схожесть. Порог `min_score=0.2` имеет разный смысл в разных endpoints. Веса рёбер графа несогласованны.

#### [M-6] Градиентная заливка не реализована
**Файл:** `frontend/components/GraphView.tsx:133-143`  
Вторичный уровень → только цвет бордера. Cytoscape не поддерживает CSS-градиенты нативно. Требование ТЗ п.7.4 (градиентная заливка при ≥2 уровнях выше порога) не выполнено.

#### [M-7] Фильтры по уровням Блума не применяются к вкладке «Анализ»
**Файл:** `frontend/pages/index.tsx:925`  
Фильтр-чекбоксы влияют только на граф. В гриде узлов на вкладке «Анализ» все узлы видны независимо от выбранных уровней.

#### [M-8] Detail-карточка узла сдвигает лейаут страницы
При наведении на узел карточка появляется ниже canvas и пушит контент вниз. Нет кнопки закрытия; `selectedNode` не сбрасывается. Пользователь не может убрать карточку.

#### [M-9] «Поиск узлов» — это `includes()`, не семантический поиск
**Файл:** `frontend/pages/index.tsx:859`  
Загружает max 200 узлов и фильтрует client-side по подстроке. При датасете > 200 узлов результаты молчаливо неполные.

#### [M-10] N+1 DB-запросы в `analyze_content`
**Файл:** `backend/app/routers/analyze.py:133`  
30 узлов = 30 отдельных SELECT перед INSERT-циклом.

#### [M-11] `annotate_dataset` — нет батчинга
**Файл:** `backend/app/tasks/tasks.py:102`  
Один INSERT на каждый chunk×level. 1000 чанков = 6000 DB round-trips до `commit()`.

#### [M-12] Нет abortController на API-вызовах
Double-click → два конкурирующих запроса. Последний выигрывает — возможен показ устаревших данных.

#### [M-13] Нет `ErrorBoundary`
Краш `GraphView` (например, некорректный `prob_vector`) → белый экран всей страницы.
